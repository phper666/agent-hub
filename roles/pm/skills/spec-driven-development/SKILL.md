# 规格驱动开发（基于 spec-kit 112k stars）

> 来源：https://github.com/github/spec-kit
> 适用于 PM 角色

## 概述

规格驱动开发（Spec-Driven Development）改变了传统开发方式：**规格成为可执行的**，直接生成工作实现，而不是仅仅指导实现。

**核心理念：** 先想清楚"做什么"和"为什么"，再决定"怎么做"。

---

## 核心原则

### 1. 规格优先

- 规格是第一等公民，不是可有可无的文档
- 规格应该足够清晰，可以直接生成代码
- 避免"vibe coding"（随意编码）

### 2. 产品场景驱动

- 关注产品场景和可预测的结果
- 不是从零开始编码每个部分
- 复用已有的模式和最佳实践

### 3. 渐进式披露

- 信息按需提供，不过早暴露细节
- 先定义高层规格，再逐步细化
- 避免信息过载

## 开发阶段

### 阶段 1：Constitution（项目原则）

定义项目的治理原则和开发指南：

```markdown
## 项目原则

### 代码质量
- 测试覆盖率 ≥ 80%
- 所有 PR 必须通过代码审查
- 遵循 SOLID 原则

### 用户体验
- 移动端优先
- 无障碍访问（WCAG 2.1 AA）
- 响应时间 < 200ms

### 安全性
- 所有输入必须验证
- 敏感数据加密存储
- 遵循最小权限原则
```

### 阶段 2：Specify（产品规格）

描述要构建什么，关注"做什么"和"为什么"：

```markdown
## 产品规格：用户认证系统

### 目标用户
- 新注册用户
- 忘记密码的用户
- 需要安全登录的用户

### 核心场景
1. **用户注册**
   - 用户输入邮箱和密码
   - 系统验证邮箱格式
   - 系统发送验证邮件
   - 用户点击链接完成注册

2. **用户登录**
   - 用户输入邮箱和密码
   - 系统验证凭据
   - 系统生成 JWT Token
   - 用户获得访问权限

3. **忘记密码**
   - 用户输入邮箱
   - 系统发送重置链接
   - 用户设置新密码
   - 系统确认修改成功

### 验收标准
- [ ] 注册成功后跳转到首页
- [ ] 登录失败显示错误提示
- [ ] 密码重置链接 24 小时内有效
- [ ] 所有操作记录审计日志
```

### 阶段 3：Plan（实现计划）

将规格拆分为可执行的实现步骤：

```markdown
## 实现计划

### 步骤 1：数据库设计
- 用户表（users）
- 验证码表（verification_codes）
- 审计日志表（audit_logs）

### 步骤 2：API 设计
- POST /api/auth/register
- POST /api/auth/login
- POST /api/auth/forgot-password
- POST /api/auth/reset-password

### 步骤 3：前端实现
- 注册页面组件
- 登录页面组件
- 忘记密码页面组件

### 步骤 4：测试
- 单元测试
- 集成测试
- E2E 测试
```

### 阶段 4：Tasks（任务拆分）

将计划拆分为具体的开发任务：

```markdown
## 任务列表

### 数据库任务
- [ ] 创建用户表迁移脚本
- [ ] 创建验证码表迁移脚本
- [ ] 创建审计日志表迁移脚本

### API 任务
- [ ] 实现注册接口
- [ ] 实现登录接口
- [ ] 实现忘记密码接口
- [ ] 实现重置密码接口

### 前端任务
- [ ] 创建注册页面
- [ ] 创建登录页面
- [ ] 创建忘记密码页面

### 测试任务
- [ ] 编写注册接口测试
- [ ] 编写登录接口测试
- [ ] 编写 E2E 测试
```

## 规格编写规范

### 好的规格

```markdown
## 用户注册

### 输入
- email: string (必填，有效邮箱格式)
- password: string (必填，至少 8 位，包含大小写字母和数字)

### 处理
1. 验证邮箱格式
2. 检查邮箱是否已注册
3. 密码加密存储
4. 生成验证令牌
5. 发送验证邮件

### 输出
- 成功：返回用户 ID，跳转到验证页面
- 失败：返回错误信息（邮箱已存在/格式错误）

### 验收标准
- [ ] 邮箱格式错误时显示提示
- [ ] 邮箱已存在时显示提示
- [ ] 注册成功后发送验证邮件
- [ ] 验证邮件 24 小时内有效
```

### 差的规格

```markdown
## 用户注册

用户可以注册账号，输入邮箱和密码就行。
```

**问题**：
- ❌ 没有输入格式要求
- ❌ 没有处理逻辑
- ❌ 没有输出定义
- ❌ 没有验收标准

## 与 Agent Hub 集成

### PM 角色使用

1. **定义项目原则**（Constitution）
   - 代码质量标准
   - 用户体验要求
   - 安全性要求

2. **编写产品规格**（Specify）
   - 目标用户
   - 核心场景
   - 验收标准

3. **创建实现计划**（Plan）
   - 数据库设计
   - API 设计
   - 前端设计

4. **拆分开发任务**（Tasks）
   - 具体的可执行任务
   - 每个任务有明确的验收标准

### 下游角色使用

- **Designer**：根据产品规格设计界面
- **Backend**：根据实现计划开发 API
- **Frontend**：根据实现计划开发前端
- **QA**：根据验收标准编写测试

## 最佳实践

### DO（推荐）

- ✅ 先定义规格，再开始编码
- ✅ 规格要足够清晰，可以生成代码
- ✅ 包含验收标准，可以验证完成
- ✅ 关注产品场景，不只是技术实现

### DON'T（避免）

- ❌ 边写代码边改需求
- ❌ 规格模糊，无法验证
- ❌ 只关注技术，不关注用户价值
- ❌ 跳过规格，直接编码
