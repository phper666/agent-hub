# 子智能体驱动开发

> 来源：superpowers（159k+ stars）
> 适用于 Frontend、Backend 角色

## 概述

通过为每个任务分派一个全新的子智能体来执行计划，每个任务完成后进行两阶段审查：先审查规格合规性，再审查代码质量。

**核心原则：** 每个任务一个全新子智能体 + 两阶段审查（先规格后质量）= 高质量、快速迭代

## 何时使用

- 有实现计划
- 任务基本独立
- 需要在当前会话中执行

## 流程

### 1. 准备阶段
- 读取计划，提取所有任务的完整文本
- 记录上下文
- 创建 TodoWrite 任务列表

### 2. 执行阶段（每个任务）

1. **分派实现子智能体**
   - 提供完整任务文本 + 上下文
   - 子智能体实现、测试、提交、自审

2. **规格合规审查**
   - 检查代码是否匹配规格
   - 发现问题 → 实现者修复 → 再次审查

3. **代码质量审查**
   - 检查代码质量
   - 发现问题 → 实现者修复 → 再次审查

4. **标记完成**
   - 在 TodoWrite 中标记任务完成

### 3. 收尾阶段
- 分派最终代码审查
- 确认所有任务完成

## 模型选择

**机械性实现任务**（1-2 个文件，清晰规格）：使用快速、便宜的模型

**集成和判断类任务**（多文件协调）：使用标准模型

**架构、设计和审查类任务**：使用最强的可用模型

## 处理实现者状态

**DONE：** 进入规格合规性审查

**DONE_WITH_CONCERNS：** 阅读疑虑，如果涉及正确性或范围，先解决

**NEEDS_CONTEXT：** 提供缺失的上下文并重新分派

**BLOCKED：** 评估阻塞原因，提供更多上下文或用更强模型重新分派

## 红线

**绝不：**
- 跳过审查（规格合规性或代码质量）
- 带着未修复的问题继续
- 并行分派多个实现子智能体（会冲突）
- 让子智能体读取计划文件（应提供完整文本）
- 忽视子智能体的问题
- 在规格合规性上接受"差不多就行"
- 跳过审查循环

## 集成

**必需的工作流：**
- **test-driven-development** - 子智能体对每个任务遵循 TDD
- **systematic-debugging** - 遇到问题时使用
- **verification-before-completion** - 完成前验证
