# 系统化调试

> 来源：superpowers（159k+ stars）
> 适用于 QA、Frontend、Backend 角色

## 概述

随意修复既浪费时间又会引入新 bug。草率的补丁只会掩盖深层问题。

**核心原则：** 在尝试修复之前，务必先找到根本原因。只修症状就是失败。

## 铁律

不做根因调查，不许提修复方案

## 何时使用

用于任何技术问题：
- 测试失败
- 生产环境 bug
- 异常行为
- 性能问题
- 构建失败
- 集成问题

## 四个阶段

你必须完成每个阶段后才能进入下一个。

### 第一阶段：根因调查

在尝试任何修复之前：

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 完整阅读堆栈跟踪
   - 记下行号、文件路径、错误码

2. **稳定复现**
   - 你能可靠地触发它吗？
   - 具体的复现步骤是什么？
   - 每次都能复现吗？

3. **检查近期变更**
   - 什么变更可能导致了这个问题？
   - git diff、最近的提交
   - 新依赖、配置变更

4. **跟踪数据流**
   - 错误值从哪里产生的？
   - 谁用错误值调用了这里？
   - 持续向上追踪直到找到源头

### 第二阶段：模式分析

先找到模式，再修复：

1. **找到可正常工作的示例**
2. **与参考实现对比**
3. **识别差异**
4. **理解依赖关系**

### 第三阶段：假设与验证

**科学方法：**

1. **提出单一假设**
   - 清晰地陈述："我认为 X 是根本原因，因为 Y"

2. **最小化测试**
   - 做出最小的改动来验证假设
   - 每次只改一个变量

3. **继续之前先验证**
   - 生效了？是 → 进入第四阶段
   - 没生效？提出新假设

### 第四阶段：实施

修复根本原因，而非症状：

1. **创建失败的测试用例**
   - 最简化的复现
   - 修复前必须先有测试

2. **实施单一修复**
   - 修复已定位的根本原因
   - 每次只改一处

3. **验证修复**
   - 测试现在通过了吗？
   - 其他测试没有被破坏吧？

4. **如果 3 次以上修复都失败了：质疑架构**
   - 停下来质疑根本性问题
   - 应该重构架构还是继续修补症状？

## 调试原则

- 不猜测，调查
- 不要同时改变多件事
- 测量，不要假设
- 一次一个变量
- 写下来
- 测试你的修复
- 承认"我不知道"

## 最终检查

- [ ] 根本原因已被识别并记录
- [ ] 修复针对的是根本原因，而非症状
- [ ] 验证测试证明修复有效
- [ ] 没有引入新问题

## Bug 修复闭环 (来自 Rules 2.1 bug-fix)

> 将调试结果落地到 Issue → 分支 → 修复 → 验证 → 提交的完整工程流程

### 1. 创建 Issue

```markdown
**Bug标题**: [组件/模块] 简短描述性标题

**环境信息**:
- OS: [macOS 14 / Windows 11 / Ubuntu 22.04]
- 应用版本: [v2.1.0]
- 运行时版本: [Node 20 / Python 3.12 / Java 21]

**重现步骤**:
1. [精确步骤 1]
2. [精确步骤 2]
3. [精确步骤 3]

**预期行为**: [应该发生什么]
**实际行为**: [实际发生了什么]

**错误日志**:
` ` `
[完整错误堆栈 + 关键上下文]
` ` `

**优先级**: High / Medium / Low
**标签**: bug, [模块名]
```

### 2. 分支与修复

```bash
# 从 main 切出 fix 分支
git checkout -b fix/issue-<编号>-<简短描述>

# 修复后运行完整测试套件
npm test  # 或 pytest / go test ./...
```

### 3. 提交

```bash
git commit -m "🐛 fix(<scope>): <简短描述> (#<issue编号>)

- 修复: <根本原因一句话>
- 新增: <如果有边界测试>
- 回归检查: 全部通过

Fixes #<issue编号>"
```

### 4. 调试记录模板

```markdown
## 调试记录: [Bug 标题]

**发现时间**: YYYY-MM-DD
**发现人**: @username
**严重程度**: Critical / High / Medium / Low

### 现象
[用户可见的异常表现]

### 根因
[经过系统化调试 4 阶段后确认的根本原因]

### 修复
[最小改动方案 + 代码 diff]

### 验证
- [ ] 修复后原 Bug 不再复现
- [ ] 新增回归测试覆盖
- [ ] 全量测试通过
- [ ] 相关角色已确认
```
