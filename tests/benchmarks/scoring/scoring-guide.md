# Agent Hub 基准测试评分指南

## 评分流程

1. 获得两份匿名输出（标记为 A 和 B）
2. 在四维度上分别打分（1-5）
3. 计算加权总分：完整性×0.3 + 技术准确性×0.3 + 格式规范性×0.2 + 实用性×0.2
4. 记录到评分模板
5. 只有两份都评完后，才揭晓哪个是 baseline、哪个是 with-agent-hub

## 详细评分标准

### 完整性 (Completeness) — 权重 0.3

| 分数 | 标准 | 示例 |
|:---:|------|------|
| 1 | 缺失 2+ 个主要章节，内容严重不完整 | PRD 只有功能描述，没有验收标准、边界条件 |
| 2 | 缺失 1 个主要章节 | PRD 缺少边界条件分析 |
| 3 | 包含所有标准章节，内容基本完整 | 每个章节都有实质性内容 |
| 4 | 所有标准章节 + 1-2 个扩展章节 | 还包含了竞品分析、性能指标 |
| 5 | 超出预期，考虑了上下游角色的使用场景 | PRD 让 Designer 可以直接开始设计，Backend 可以直接开始建模 |

### 技术准确性 (Technical Accuracy) — 权重 0.3

| 分数 | 标准 | 示例 |
|:---:|------|------|
| 1 | 包含明显的事实性错误 | 声称"JWT 适合存储大量用户数据" |
| 2 | 有 1-2 处技术疑点 | SQL 语句有索引设计不合理 |
| 3 | 技术上基本正确 | API 设计合理，DDL 语法正确 |
| 4 | 技术方案经过推敲，考虑了常见的坑 | 考虑了 N+1 问题、缓存策略 |
| 5 | 专家级方案，包含 trade-off 分析和替代方案 | 解释了为什么选 Redis 而不是数据库做购物车缓存 |

### 格式规范性 (Format Quality) — 权重 0.2

| 分数 | 标准 | 示例 |
|:---:|------|------|
| 1 | 纯文本无结构 | 一大段文字没有标题、列表 |
| 2 | 有基本结构但不一致 | 有的章节用标题，有的用加粗 |
| 3 | 遵循标准格式 | Markdown 标题层次正确，表格格式规范 |
| 4 | 格式专业，可读性强 | 使用了 Mermaid 图表、代码块高亮 |
| 5 | 格式可以直接作为团队模板使用 | 其他人可以照着这个格式写自己的文档 |

### 实用性 (Practicality) — 权重 0.2

| 分数 | 标准 | 示例 |
|:---:|------|------|
| 1 | 无法执行 | PRD 没有具体的功能列表 |
| 2 | 需要大量修改才能使用 | 只有框架，内容需要重新填充 |
| 3 | 稍作修改即可执行 | 验收标准需要更具体一点 |
| 4 | 可直接使用 | PRD 可以发给开发团队开始实现 |
| 5 | 开箱即用 + 考虑了下游 | Backend 可以直接按 API Spec 写代码 |

## 计算示例

```
Baseline 评分: 完整性 3 + 技术准确性 2 + 格式规范性 3 + 实用性 2
Baseline 加权分: 3×0.3 + 2×0.3 + 3×0.2 + 2×0.2 = 2.5 / 5

With Agent Hub: 完整性 4 + 技术准确性 4 + 格式规范性 4 + 实用性 4
With Agent Hub 加权分: 4×0.3 + 4×0.3 + 4×0.2 + 4×0.2 = 4.0 / 5

提升: (4.0 - 2.5) / 2.5 = +60%
```
