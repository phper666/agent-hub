## Output Rules（所有角色的输出规范）

### 核心原则

1. **Concise** — 能用一句话说清的不用一段话
2. **Structured** — 使用标题、列表、表格组织信息，不写大段纯文本
3. **Actionable** — 每条输出都应指向明确的下一步动作

### 回复格式

4. 回复开头用一句话概括结论或建议，再展开细节（倒金字塔结构）
5. 需要用户决策时，用编号列表给出选项，推荐项放在第一位
6. 展示代码时必须标注语言类型（` ```python `），并附带必要上下文
7. 超过 5 步的操作流程使用编号列表，不超过 5 步可用段落

### 文件输出

8. 生成的文档必须包含：标题、日期、版本号（v1.0）
9. Markdown 文件遵循 [Google Markdown Style Guide](https://google.github.io/styleguide/docguide/style.html)
10. 每个输出文件顶部 3 行说明该文件的用途、消费者和更新频率
11. 代码文件头部注释说明模块职责，格式: `// <module>: <one-line purpose>`

### 错误与边界情况

12. 发现输入不完整或有歧义时，先列出已知信息和缺失信息，再提问
13. 不能完成任务时，明确说明原因 + 建议的替代方案
14. 不确定的信息标注 `[待确认]`，不假装确定
15. 报告 bug 时格式: 现象 → 复现步骤 → 预期行为 → 实际行为 → 建议修复方向

### 交互规范

16. 一次只问一个问题，避免一次性抛出多个问题让用户应接不暇
17. 用户说"继续"时从上次中断点继续，不重复已完成的部分
18. 长任务每完成一个阶段主动汇报进度（"Step 2/5 ✅: 数据库设计已完成"）
19. 术语首次出现时给出简短解释或链接

### 状态更新

20. 更新 `docs/current/status.md` 时格式一致：状态 emoji + 文字 + 时间戳
21. 状态值规范：
    - `⏳ 待开始` — 尚未开始
    - `🔄 进行中` — 正在执行
    - `✅ 完成` — 已完成且验证通过
    - `❌ 阻塞` — 被阻塞，注明原因
22. 交接给下游角色时，列出已完成内容和待处理事项
