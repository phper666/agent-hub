## Output Rules（所有角色的输出规范）

### 核心原则

1. **Concise** — 能用一句话说清的不用一段话
2. **Structured** — 使用标题、列表、表格组织信息，不写大段纯文本
3. **Actionable** — 每条输出都应指向明确的下一步动作

### 回复格式

4. 回复开头用一句话概括结论或建议，再展开细节（倒金字塔结构）
5. 需要用户决策时，用编号列表给出选项，推荐项放在第一位
6. 展示代码时必须标注语言类型（` ```python `），并附带必要上下文
7. 超过 5 步的操作流程使用编号列表，不超过 5 步可用段落

### 文件输出

8. 生成的文档必须包含：标题、日期、版本号（v1.0）
9. Markdown 文件遵循 [Google Markdown Style Guide](https://google.github.io/styleguide/docguide/style.html)
10. 每个输出文件顶部 3 行说明该文件的用途、消费者和更新频率
11. 代码文件头部注释说明模块职责，格式: `// <module>: <one-line purpose>`

### 错误与边界情况

12. 发现输入不完整或有歧义时，先列出已知信息和缺失信息，再提问
13. 不能完成任务时，明确说明原因 + 建议的替代方案
14. 不确定的信息标注 `[待确认]`，不假装确定
15. 报告 bug 时格式: 现象 → 复现步骤 → 预期行为 → 实际行为 → 建议修复方向

### 交互规范

16. 一次只问一个问题，避免一次性抛出多个问题让用户应接不暇
17. 用户说"继续"时从上次中断点继续，不重复已完成的部分
18. 长任务每完成一个阶段主动汇报进度（"Step 2/5 ✅: 数据库设计已完成"）
19. 术语首次出现时给出简短解释或链接

### 状态更新

20. 更新 `docs/current/status.md` 时格式一致：状态 emoji + 文字 + 时间戳
21. 状态值规范：
    - `⏳ 待开始` — 尚未开始
    - `🔄 进行中` — 正在执行
    - `✅ 完成` — 已完成且验证通过
    - `❌ 阻塞` — 被阻塞，注明原因
22. 交接给下游角色时，列出已完成内容和待处理事项

### 精简输出规范（基于 headroom 28k stars）

> 来源：https://github.com/chopratejas/headroom
> 目标：60-95% fewer tokens，same answers

#### Token 优化原则

1. **压缩优先** — 在输出前先压缩内容，保留核心信息
2. **结构化输出** — 使用表格、列表、代码块，避免大段纯文本
3. **去除冗余** — 删除重复信息、无用的修饰词、过长的解释
4. **关键信息前置** — 最重要的信息放在最前面

#### 输出压缩技巧

| 场景 | 压缩方法 | 示例 |
|------|----------|------|
| 代码搜索结果 | 只保留文件名、行号、关键代码 | ❌ 完整文件内容 ✅ `file.ts:42: const x = 1` |
| 日志输出 | 提取错误信息，去除调试信息 | ❌ 100 行日志 ✅ `Error: ENOENT: no such file` |
| 文档内容 | 提取标题、关键点、结论 | ❌ 完整文档 ✅ `## 核心要点\n- 要点1\n- 要点2` |
| API 响应 | 只保留状态码、关键字段 | ❌ 完整 JSON ✅ `200 OK: { id, name, status }` |

#### 状态报告格式

```
## 进度报告

**当前阶段**: Step 3/5
**已完成**: ✅ 数据库设计, ✅ API 设计
**进行中**: 🔄 前端组件开发
**待处理**: ⏳ 测试, ⏳ 部署

**下一步**: 继续前端组件开发
```

#### 代码输出规范

1. **只展示关键代码** — 不展示完整的文件内容，只展示修改的部分
2. **附带上下文** — 说明代码的作用、位置、修改原因
3. **使用 diff 格式** — 展示修改前后的对比

```diff
// file: src/utils/helper.ts
// 作用: 添加日期格式化函数
+ export function formatDate(date: Date): string {
+   return date.toISOString().split('T')[0];
+ }
```

#### 文档输出规范

1. **标题层级清晰** — 使用 H1-H3，不超过 3 级
2. **要点列表化** — 每个要点一行，不超过 25 字
3. **表格化数据** — 结构化数据使用表格
4. **代码块标注语言** — 所有代码块必须标注语言类型

#### 交互优化

1. **一次一个问题** — 避免一次性抛出多个问题
2. **选项编号** — 需要用户选择时，用编号列表
3. **进度汇报** — 长任务每完成一个阶段主动汇报
4. **术语解释** — 首次出现的术语给出简短解释

#### Token 节省检查清单

在输出前检查：

- [ ] **去除重复** — 是否有重复的信息？
- [ ] **精简语言** — 是否有冗余的修饰词？
- [ ] **结构化** — 是否使用了合适的格式（表格/列表/代码块）？
- [ ] **关键信息前置** — 最重要的信息是否在最前面？
- [ ] **代码压缩** — 是否只展示了关键代码？
- [ ] **日志压缩** — 是否提取了关键日志信息？
