## Quality Rules（所有角色的质量底线）

### 测试质量

1. 测试覆盖率 ≥ 80%（行覆盖 + 分支覆盖）
2. 新增代码必须有对应测试，且测试必须在同一次 PR 中提交
3. 单元测试必须独立运行，不依赖外部服务或执行顺序
4. 测试命名清晰：`test_<被测行为>_<条件>_<预期结果>`
5. 测试必须包含 happy path、edge case 和 error case

### 代码审查

6. 所有 Critical 级别的 AI review 问题必须修复后才能合并
7. 所有 Warning 级别问题必须逐条确认（修复或注明原因后跳过）
8. 代码审查关注：正确性 > 安全性 > 可读性 > 性能

### 代码规范

9. 代码不得包含硬编码的密钥、Token、密码
10. 日志中不得出现 PII（手机号、邮箱、身份证、密码）
11. 所有公开方法必须有空值校验（防御性编程）
12. 所有异步操作必须有错误处理（try/catch 或 .catch）
13. 数据库写操作必须在事务中
14. 代码文件不超过 300 行，超过则拆分
15. 函数不超过 50 行，超过则重构
16. 函数参数不超过 4 个，超过则使用 options 对象
17. 嵌套层级不超过 3 层，超过则提取函数或使用 early return

### 可维护性

18. 命名自解释——变量名应说明意图，不使用缩写（除广泛接受的如 `id`, `url`, `api`）
19. 注释只解释 **why**，不解释 **what**（代码本身应说明 what）
20. 消除 magic number / magic string，提取为命名常量
21. 遵循 DRY 原则——重复超过 2 次的逻辑必须抽取
22. 遵循 SOLID 原则——尤其是单一职责和依赖倒置

### 文档

23. 所有公开 API 必须有接口文档（参数、返回值、错误码）
24. 复杂算法或业务逻辑必须附带行内注释说明思路
25. README 必须包含：项目简介、启动方式、环境变量说明
26. 架构决策记录在 `docs/architecture/decisions/` 中（ADR 格式）

### 性能基线

27. API 响应时间 P95 < 500ms（可按业务调整）
28. 前端首屏加载 < 3s（LCP）
29. 数据库查询必须有索引覆盖，禁止全表扫描（除明确允许的小表）
30. N+1 查询必须消除（使用 eager load / batch query）
